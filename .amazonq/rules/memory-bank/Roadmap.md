---

1. Допущения и вопросы к заказчику (статус: согласовано)

Принятые допущения

1. Монорепозиторий: Вся разработка ведётся на TypeScript в едином проекте расширения VS Code.
2. Webview для чата: Интерфейс чата реализуется через Webview, что даёт полный контроль над UI (аналогично GitHub Copilot Chat).
3. AST-парсинг: Для анализа кода используется Tree-sitter (WASM) с фоллбэком на регулярные выражения для сложных языков.
4. LLM-провайдер: В качестве основного интерфейса используется OpenAI-совместимый API (поддерживает прокси, Ollama, прямые сервисы).
5. VS Code Stable API: Используется исключительно стабильное API VS Code, без внутренних методов.
6. Файловая структура: Roadmap и чек-лист сохраняются в папку .devil/ (утверждено заказчиком для порядка).

Вопросы, получившие ответ

1. Метод шифрования: Используем библиотеку keytar для безопасного хранения API-ключей в Credential Manager ОС (Windows 11).
2. Порог фоновой задачи: Задача, выполняющаяся дольше 15 секунд, переводится в фоновый режим с уведомлением в VS Code.
3. Приоритет индексации: Модуль быстрого поиска должен быть реализован в первой версии (Must have). Срок не критичен, что учтено в новом спринте 4.
4. Расположение Roadmap и чек-листа: Все артефакты проекта хранятся в .devil/, корень проекта не засоряется.

Риски и блокеры:

· Технический долг: Tree-sitter может потребовать нативной компиляции. Решение: использовать WASM-сборку или перейти на RegExp в случае проблем.
· Зависимость: Сложность парсинга Python. Решение: фокус на JS/TS, Python — только импорты и определения верхнего уровня.

Глобальная цель: Реализовать систему индексации кода для мгновенного полнотекстового и семантического поиска, а также статический анализ по запросу.

Задачи на фронтенд:

· UI-13: Реализовать отображение результатов поиска в чате в виде кликабельного списка с путями и сниппетами кода. ТЗ: Модуль 5.

DevOps / Инфраструктурные задачи:

· DEVOPS-09: Провести нагрузочное тестирование индексации на проекте из 1500 файлов (построение индекса <5 сек, поиск <200 мс).

Чекпоинты (Checkpoints):

· Команда /search useEffect в React-проекте возвращает все файлы, содержащие useEffect, за время <0.5 сек.
· Команда /semsearch "функция для отправки запроса" возвращает файл с fetchData, даже если слово "запрос" в коде не встречается.
· Команда /lint запускает ESLint и показывает реальные ошибки с возможностью перехода по клику.
· Индекс обновляется при сохранении файла (задержка не более 2 сек).

Критерии готовности (DoD):

· Поиск и индексация покрыты юнит-тестами (мок файловой системы).
· Векторное хранилище не вызывает крашей при пустом проекте или отсутствии файлов.
· Модель для embedding загружается один раз в фоновом режиме и не блокирует активацию расширения.

Риски и блокеры:

· Производительность: Embedding-модель может потреблять много памяти и процессора. Решение: использовать квантованную модель (INT8) и ограничить размер индексируемых файлов до 10 КБ на файл.
· Совместимость линтеров: ESLint должен быть установлен в проекте пользователя. Решение: проверять наличие конфига, иначе выдавать инструкцию по установке.
