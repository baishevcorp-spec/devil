## 🎯 Ключевые принятые решения

| Вопрос | Решение | Обоснование |
|--------|---------|-------------|
| Когда запускать Extract? | **Вариант B** — только по триггерам (создание файла, `/remember`, принятие решения) | Экономия токенов LLM (автоматический Extract после каждого ответа удваивает стоимость API) |
| Где хранить `Why` / `How to apply`? | **Вариант A** — в `metadata` JSON существующих узлов графа | Не требует изменения схемы БД, использует существующую инфраструктуру |
| Когда запускать Dream? | **Гибрид** — раз в сутки + вручную через `/dream` + при >100 изменениях за день | Баланс между актуальностью памяти и производительностью |
| Формат хранения памяти | SQLite (как в основной схеме Devil) | Отказ от YAML+Markdown в пользу единой БД |
| Векторное хранилище | SQLite + transformers.js (BCK-29) | Отказ от Chroma в пользу единой БД |

## ✅ Критерии успеха (новые)

### Критерий 5: Качество памяти

После запуска `/dream` не остаётся "висячих" связей (узлы без связей или связи без узлов).

## ⚠️ Риски и митигация

| Риск | Вероятность | Влияние | Меры смягчения |
|------|-------------|---------|----------------|
| Автоматический Extract генерирует некачественные узлы | Средняя | Среднее | Триггеры только для ключевых событий, валидация через LLM |
| Dream удаляет важные узлы | Низкая | Высокое | Логирование всех удалений в `change_log`, возможность отката |
| Dream выполняется слишком долго | Средняя | Низкое | Лимит: не более 5 минут, прерывание по таймауту |
| Recall перегружает контекст LLM | Средняя | Среднее | Жёсткий лимит 5 узлов, приоритет по релевантности |
| Файловая блокировка Dream не освобождается | Низкая | Низкое | Автоматическое удаление блокировки старше 1 часа |
