## 1. Обоснование изменений

На основе анализа архитектуры Managed Auto-Memory от Qwen Code и существующей графовой памяти Devil (SQLite), предлагается внедрить систему управления жизненным циклом памяти через 4 операции:

| Операция | Назначение | Аналог в Qwen |
|----------|------------|---------------|
| **Extract** | Извлечение знаний из диалогов и действий | Extract |
| **Dream** | Фоновая интеграция и дедупликация | Dream |
| **Recall** | Поиск релевантной памяти перед запросом | Recall |
| **Forget** | Ручное удаление конкретных записей | Forget |

Это превращает Devil из "пассивного хранилища" в "активную систему памяти", которая:
- Автоматически накапливает знания
- Поддерживает целостность графа
- Предоставляет релевантный контекст LLM
- Позволяет пользователю управлять памятью

## 2. Принятые решения

### Решение 1: Когда запускать Extract?
**Выбран вариант B** — запуск только по триггерам (не после каждого ответа).

**Триггеры:**
- Создание/обновление файла через `/dev next`
- Принятие архитектурного решения (через `/remember` или автоматически при ключевых словах)
- Явная команда пользователя `/remember [текст]`
- Завершение этапа Roadmap

**Обоснование:** Экономия токенов LLM (автоматический Extract после каждого ответа удваивает стоимость API).

### Решение 2: Где хранить Why/How to apply?
**Выбран вариант A** — в `metadata` JSON существующих узлов графа.

**Формат:**
```json
{
  "line": 42,
  "why": "Избегаем race conditions при параллельных запросах",
  "how_to_apply": "Использовать только в модулях работы с БД"
}
```

**Обоснование:** Не требует изменения схемы БД, использует существующую инфраструктуру.

### Решение 3: Когда запускать Dream?
**Выбран гибрид** — комбинация подходов:
- Раз в сутки (автоматически, в фоновом режиме)
- Вручную через команду `/dream`
- Автоматически при >100 изменениях за день

**Обоснование:** Баланс между актуальностью памяти и производительностью.

**Ответственность:**

### Новые команды

#### Команда `/forget [id]`
**Приоритет:** Should have
**Время:** ~1 час

**Описание:** Точное удаление конкретной записи из всех хранилищ (граф, векторный индекс, профиль).

**Логика:**
1. Парсинг ID из аргумента команды
2. Удаление узла из `nodes` (каскадное удаление связей через FOREIGN KEY)
3. Удаление embedding из векторного индекса
4. Если это `custom_instruction` — удаление из `user_profile`
5. Логирование в `change_log` (action='forget')
6. Уведомление пользователя: "Удалена запись: [название]"

**Обработка ошибок:**
- Если ID не найден: "Запись с ID [id] не существует"
- Если ID защищённый (системный узел): "Эта запись не может быть удалена"

---

#### Команда `/dream`
**Приоритет:** Should have
**Время:** ~1 час

**Описание:** Ручной запуск фоновой задачи Dream (дедупликация, интеграция, валидация).

**Логика:**
1. Проверка блокировки `.devil/.dream.lock`
2. Если Dream уже запущен: "Dream уже выполняется. Подождите завершения."
3. Создание блокировки
4. Запуск `DreamManager.runDream()` в фоне (через `vscode.window.withProgress`)
5. Показ прогресса: "Оптимизация памяти... [процент]"
6. По завершению: уведомление с отчётом (сколько дедуплицировано, удалено, консолидировано)
7. Удаление блокировки

**UI:**
- Иконка ✦ в статус-баре во время выполнения
- Уведомление VS Code по завершению

---

#### Команда `/memory export`
**Приоритет:** Could have
**Время:** ~1 час

**Описание:** Экспорт графовой памяти в человекочитаемый формат Markdown.

**Логика:**
1. Чтение всех узлов из `nodes` (группировка по типу)
2. Чтение всех связей из `edges`
3. Генерация `.devil/MEMORY.md` с разделами:
   - `## Файлы` (узлы type='file')
   - `## Классы` (узлы type='class')
   - `## Функции` (узлы type='function')
   - `## Решения` (узлы type='decision')
   - `## Технологии` (узлы type='technology')
   - `## Связи` (список всех связей)
4. Для каждого узла с `why` и `how_to_apply` — добавление этих полей

**Формат:**
```markdown
# Память проекта Devil

## Решения

### Использовать React для UI
- **Путь:** src/components/App.tsx
- **Почему:** Компонентный подход упрощает переиспользование
- **Как применять:** Все новые компоненты — функциональные, с хуками

## Связи

- `App.tsx` → импортирует → `Button.tsx`
- `handleSubmit` → вызывает → `validateForm`
```

## 5. Риски и митигация

| Риск | Вероятность | Влияние | Меры смягчения |
|------|-------------|---------|----------------|
| Автоматический Extract генерирует некачественные узлы | Средняя | Среднее | Триггеры только для ключевых событий, валидация через LLM |
| Dream удаляет важные узлы | Низкая | Высокое | Логирование всех удалений в `change_log`, возможность отката |
| Dream выполняется слишком долго | Средняя | Низкое | Лимит: не более 5 минут, прерывание по таймауту |
| Recall перегружает контекст LLM | Средняя | Среднее | Жёсткий лимит 5 узлов, приоритет по релевантности |
| Файловая блокировка Dream не освобождается | Низкая | Низкое | Автоматическое удаление блокировки старше 1 часа |

## 6. Критерии успеха (дополнительные)

1. **Качество памяти:** После 10 диалогов с решениями, граф содержит узлы `decision` с полями `why` и `how_to_apply`.
2. **Целостность графа:** После запуска `/dream` не остаётся "висячих" связей (узлы без связей или связи без узлов).
3. **Релевантность Recall:** При запросе "как реализовать аутентификацию?" агент предлагает решения из памяти, если они есть.
4. **Производительность Dream:** Дедупликация 1000 узлов занимает не более 30 секунд.
5. **Удобство управления:** Команда `/forget` удаляет запись за <1 секунду, `/dream` показывает прогресс.
