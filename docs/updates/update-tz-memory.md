## 1. Обоснование изменений

На основе анализа архитектуры Managed Auto-Memory от Qwen Code и существующей графовой памяти Devil (SQLite), предлагается внедрить систему управления жизненным циклом памяти через 4 операции:

| Операция | Назначение | Аналог в Qwen |
|----------|------------|---------------|
| **Extract** | Извлечение знаний из диалогов и действий | Extract |
| **Dream** | Фоновая интеграция и дедупликация | Dream |
| **Recall** | Поиск релевантной памяти перед запросом | Recall |
| **Forget** | Ручное удаление конкретных записей | Forget |

Это превращает Devil из "пассивного хранилища" в "активную систему памяти", которая:
- Автоматически накапливает знания
- Поддерживает целостность графа
- Предоставляет релевантный контекст LLM
- Позволяет пользователю управлять памятью

## 2. Принятые решения

### Решение 1: Когда запускать Extract?
**Выбран вариант B** — запуск только по триггерам (не после каждого ответа).

**Триггеры:**
- Создание/обновление файла через `/dev next`
- Принятие архитектурного решения (через `/remember` или автоматически при ключевых словах)
- Явная команда пользователя `/remember [текст]`
- Завершение этапа Roadmap

**Обоснование:** Экономия токенов LLM (автоматический Extract после каждого ответа удваивает стоимость API).

### Решение 2: Где хранить Why/How to apply?
**Выбран вариант A** — в `metadata` JSON существующих узлов графа.

**Формат:**
```json
{
  "line": 42,
  "why": "Избегаем race conditions при параллельных запросах",
  "how_to_apply": "Использовать только в модулях работы с БД"
}
```

**Обоснование:** Не требует изменения схемы БД, использует существующую инфраструктуру.

### Решение 3: Когда запускать Dream?
**Выбран гибрид** — комбинация подходов:
- Раз в сутки (автоматически, в фоновом режиме)
- Вручную через команду `/dream`
- Автоматически при >100 изменениях за день

**Обоснование:** Баланс между актуальностью памяти и производительностью.

## 4. Изменения в схеме данных

### 4.1. Новый тип узла

Добавить тип `reference` в CHECK constraint таблицы `nodes`:

```sql
type TEXT NOT NULL CHECK (type IN ('file', 'class', 'function', 'variable', 'technology', 'decision', 'concept', 'reference'))
```

### 4.2. Расширение metadata

Для узлов типов `decision`, `concept`, `reference` поддерживать в metadata поля:

```json
{
  "why": "Причина принятия решения",
  "how_to_apply": "Когда и как применять это знание"
}
```

Для узлов типа `reference` дополнительно:
```json
{
  "url": "https://...",
  "description": "Описание ресурса",
  "last_accessed": 1751414400000
}
```

### 4.3. Новые действия в change_log

Добавить в CHECK constraint таблицы `change_log`:

```sql
action TEXT NOT NULL CHECK (action IN ('create', 'update', 'delete', 'scan', 'generate', 'extract', 'dream', 'recall', 'forget'))
```

## 5. Новые критерии успеха

Добавить к существующим 4 критериям из ТЗ:

### Критерий 5: Качество памяти
После 10 диалогов с решениями, граф содержит узлы `decision` с полями `why` и `how_to_apply` в metadata.

**Проверка:**
```sql
SELECT COUNT(*) FROM nodes
WHERE type = 'decision'
AND metadata LIKE '%why%'
AND metadata LIKE '%how_to_apply%';
-- Ожидаемый результат: >= 10
```

### Критерий 6: Целостность графа
После запуска `/dream` не остаётся "висячих" связей (узлы без связей или связи без узлов).

**Проверка:**
```sql
-- Связи без узлов
SELECT COUNT(*) FROM edges e
LEFT JOIN nodes n1 ON e.from_node = n1.id
LEFT JOIN nodes n2 ON e.to_node = n2.id
WHERE n1.id IS NULL OR n2.id IS NULL;
-- Ожидаемый результат: 0
```

### Критерий 7: Релевантность Recall
При запросе "как реализовать аутентификацию?" агент предлагает решения из памяти, если они есть.

**Проверка:**
- Создать узел `decision` с name="Использовать JWT для аутентификации"
- Отправить запрос "как реализовать аутентификацию?"
- Проверить, что агент упоминает JWT в ответе

## 6. Новые User Stories

Добавить к существующим 10 User Stories из ТЗ:

### User Story 11: Автоматическое извлечение решений
**Как разработчик**, я хочу, чтобы агент автоматически извлекал архитектурные решения из наших диалогов и сохранял их в памяти, чтобы не повторять одни и те же обсуждения.

**Критерии готовности:**
- При создании файла через `/dev next` агент анализирует диалог
- Если в диалоге есть решение (например, "используем React"), создаётся узел `decision`
- Узел содержит поля `why` и `how_to_apply` в metadata
- Решение появляется в `/memory show`

### User Story 12: Автоматическое извлечение инструкций
**Как разработчик**, я хочу, чтобы агент автоматически запоминал мои инструкции ("всегда используй TypeScript", "не используй any"), чтобы применять их в будущих генерациях.

**Критерии готовности:**
- При фразе "всегда делай X" или "не используй Y" в диалоге агент извлекает инструкцию
- Инструкция сохраняется в `user_profile.custom_instructions`
- При следующей генерации кода агент учитывает эту инструкцию
- Инструкция появляется в `/memory profile`

### User Story 13: Ручное сохранение знаний
**Как разработчик**, я хочу иметь возможность явно сохранить важное знание через команду `/remember`, чтобы не полагаться только на автоматическое извлечение.

**Критерии готовности:**
- Команда `/remember Используем PostgreSQL для хранения данных` создаёт узел `decision`
- Узел содержит текст из команды в поле `name`
- Узел появляется в `/memory show`
- Можно добавить `why` и `how_to_apply` через дополнительные аргументы

### User Story 14: Фоновая оптимизация памяти
**Как разработчик**, я хочу, чтобы агент автоматически оптимизировал память (удалял дубликаты, мёртвые связи), чтобы граф оставался чистым и актуальным.

**Критерии готовности:**
- Раз в сутки запускается фоновая задача Dream
- Dream удаляет дубликаты узлов (по имени + пути)
- Dream удаляет связи, указывающие на несуществующие узлы
- Dream консолидирует повторяющиеся `custom_instructions`
- Результат Dream логируется в `change_log`

### User Story 15: Ручной запуск оптимизации
**Как разработчик**, я хочу иметь возможность вручную запустить оптимизацию памяти через команду `/dream`, чтобы не ждать автоматического запуска.

**Критерии готовности:**
- Команда `/dream` запускает DreamManager.runDream()
- Во время выполнения показывается прогресс-бар в VS Code
- По завершению выводится отчёт (сколько дедуплицировано, удалено, консолидировано)
- Команда не блокирует основной поток VS Code

### User Story 16: Удаление записей из памяти
**Как разработчик**, я хочу иметь возможность удалить конкретную запись из памяти через команду `/forget`, если она устарела или неверна.

**Критерии готовности:**
- Команда `/forget [id]` удаляет узел с указанным ID
- Каскадно удаляются все связи этого узла
- Удаляется embedding из векторного индекса (если есть)
- Если это `custom_instruction` — удаляется из `user_profile`
- Операция логируется в `change_log` (action='forget')
- Если ID не найден — выводится понятная ошибка

## 7. Изменения в нефункциональных требованиях

### 7.1. Производительность

Добавить новое требование:

| Категория | Требование |
|-----------|------------|
| Производительность памяти | Dream (дедупликация 1000 узлов) занимает не более 30 секунд |
| Производительность памяти | Recall (поиск релевантных узлов) занимает не более 500 мс |
| Производительность памяти | Extract (извлечение из диалога) занимает не более 2 секунд |

### 7.2. Отказоустойчивость

Добавить новое требование:

| Категория | Требование |
|-----------|------------|
| Отказоустойчивость памяти | При ошибке Dream (например, блокировка уже существует) — пропуск запуска, логирование ошибки |
| Отказоустойчивость памяти | При ошибке Extract — продолжение работы, логирование ошибки в `change_log` |
| Отказоустойчивость памяти | При ошибке Recall — запрос к LLM без контекста памяти (фоллбэк) |

### 7.3. Безопасность

Добавить новое требование:

| Категория | Требование |
|-----------|------------|
| Безопасность памяти | Команда `/forget` не может удалить системные узлы (тип='file' для открытых файлов) |
| Безопасность памяти | Dream не удаляет узлы, созданные менее 24 часов назад |

- Чтение всех узлов из `nodes` (группировка по типу)
- Чтение всех связей из `edges`
- Генерация `.devil/MEMORY.md` с разделами по типам узлов
- Для каждого узла с `why` и `how_to_apply` — добавление этих полей

**Время:** ~1 час
