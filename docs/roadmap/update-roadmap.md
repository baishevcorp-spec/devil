## Дополнение к Roadmap (добавить в Sprint 2)

### BCK-09 (обновлённая версия): Команда `/roadmap generate` с интервью

**Текущая реализация:** Генерация roadmap.md на основе структуры проекта + интервью (interview.json)

**Новые требования:**
1. **Структурированное интервью** (interview.json вместо interview.md):
   - Поля: `projectName`, `description`, `goals`, `techStack`, `targetAudience`, `deadlines`, `constraints`
   - Валидация обязательных полей перед генерацией
   - Автоматическая проверка заполнения

2. **Стейт-машина** (`.devil/.interview_status.json`):
   ```json
   {
     "status": "pending|filled|roadmap_generated",
     "createdAt": 1234567890,
     "updatedAt": 1234567890,
     "interviewFile": "interview.json"
   }
   ```
   - `pending` — интервью создано, ждёт заполнения
   - `filled` — интервью заполнено, готово к генерации
   - `roadmap_generated` — roadmap сгенерирован

3. **Команда `/roadmap update`**:
   - Перегенерация roadmap.md с учётом новых данных
   - Сохранение истории версий (roadmap_v1.md, roadmap_v2.md)
   - Возможность отката к предыдущей версии

**Критерии готовности:**
- Интервью хранится в JSON с валидацией полей
- Статус интервью отслеживается в `.interview_status.json`
- Команда `/roadmap update` перегенерирует roadmap с сохранением истории

### BCK-10 (обновлённая версия): Команда `/checklist generate` с интеграцией roadmap

**Текущая реализация:** Простая генерация списка файлов

**Новые требования:**
1. **Интеграция с roadmap.md**:
   - Читать этапы из roadmap.md
   - Разбивать каждый этап на конкретные файлы/задачи
   - Связывать чек-лист с этапами roadmap

2. **Формат с чекбоксами**:
   ```markdown
   # Чек-лист проекта

   ## Этап 1: Инициализация проекта
   - [ ] `package.json` — конфигурация проекта и зависимости
   - [ ] `tsconfig.json` — настройки TypeScript
   - [ ] `.eslintrc.json` — правила линтера

   ## Этап 2: Базовая структура
   - [ ] `src/extension.ts` — точка входа расширения
   - [ ] `src/services/ConfigManager.ts` — управление конфигурацией
   ```

3. **Валидация путей**:
   - Проверять `fileExists()` перед записью в чек-лист
   - Помечать несуществующие файлы как `[ ]` (требуется создание)
   - Существующие файлы помечать как `[x]` (уже реализовано)

4. **Команда `/checklist sync`**:
   - Сканировать реальную структуру проекта
   - Обновлять статусы `[ ]` → `[x]` для существующих файлов
   - Добавлять новые файлы, которых нет в чек-листе
   - Генерировать отчёт: "Добавлено 5 файлов, обновлено 3 статуса"

5. **Фильтрация** (опционально):
   - `/checklist generate --scope=backend` — только backend-файлы
   - `/checklist generate --stage=phase1` — только файлы первого этапа

**Критерии готовности:**
- Чек-лист генерируется на основе roadmap.md
- Формат с чекбоксами и группировкой по этапам
- Пути валидируются через fileExists()
- Команда `/checklist sync` обновляет статусы

## 3. Архитектура `/dev generate`

### Компоненты

```
/dev generate
    ↓
DevPlanManager
    ├─ Сбор контекста (interview, roadmap, checklist, graph)
    ├─ Формирование ТЗ через LLM
    ├─ Генерация плана (список шагов)
    └─ Сохранение плана в .devil/dev-plan.json
    ↓
/dev next
    ↓
DevPlanExecutor
    ├─ Чтение текущего шага из плана
    ├─ Проверка существования файла
    ├─ Бэкап (если файл существует)
    ├─ Генерация кода через LLM
    ├─ Создание файла
    ├─ Обновление статуса в плане
    └─ Возврат результата + команды для проверки
```

### Структура плана (`.devil/dev-plan.json`)

```json
{
  "version": 1,
  "createdAt": 1234567890,
  "updatedAt": 1234567890,
  "status": "in_progress",
  "totalSteps": 15,
  "completedSteps": 3,
  "steps": [
    {
      "id": 1,
      "type": "create_directory",
      "path": "src/services",
      "description": "Создать директорию для сервисов",
      "status": "completed",
      "completedAt": 1234567890
    },
    {
      "id": 2,
      "type": "create_file",
      "path": "src/services/UserService.ts",
      "description": "Создать UserService для работы с пользователями",

```

### Команды

| Команда | Описание |
|---------|----------|
| `/dev generate` | Собрать ТЗ и план разработки |
| `/dev next` | Выполнить следующий шаг |
| `/dev skip [id]` | Пропустить шаг (опционально с ID) |
| `/dev status` | Показать прогресс |
| `/dev reset` | Сбросить план |
| `/dev show [id]` | Показать детали шага |

**Итого:** ~8 часов

### Сильные стороны

1. **Автоматизация рутины** — создание структуры проекта за 1 команду
2. **Консистентность** — все файлы создаются с учётом общего контекста
3. **Контроль** — пошаговое выполнение, можно остановиться на любом шаге
4. **Интеграция с графом** — агент "помнит", что уже создано
5. **Соответствие ТЗ** — это именно то, что описано в "Модуль 4: Генерация скрипта структуры"

### Риски

| Риск | Решение |
|------|---------|
| LLM может сгенерировать неоптимальный порядок файлов | Использовать граф зависимостей для определения порядка |
| Большой объём контекста (roadmap + checklist + граф) | Передавать только релевантные этапы |
| Пользователь может потерять контроль | Пошаговое выполнение + возможность отмены |
| Сложность реализации | Разбить на 3 этапа: ТЗ → план → выполнение |

Почему это важно:
- Это **killer feature** проекта — то, чего нет у Copilot Chat
- Превращает Devil из "чат-ассистента" в "автономного разработчика"
- Реализует ключевой критерий успеха из ТЗ: "время на вспоминание контекста сократилось на 70%"
