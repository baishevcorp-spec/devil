Техническое задание на разработку «Devil» — интеллектуального агента-ассистента для разработки с долговременной памятью

1. Общее описание

Название проекта: Devil

Краткое описание продукта:
«Devil» — это расширение для VS Code (Windows 11), представляющее собой интеллектуального агента-ассистента для разработчиков. Агент решает фундаментальную проблему потери контекста проекта на протяжении всего жизненного цикла разработки — от идеи до финального релиза. Он обладает долговременной гибридной памятью (графовая база знаний + обучение на предпочтениях пользователя), умеет читать и анализировать файлы проекта, отслеживать изменения через Git, генерировать код, патчи, тесты, объяснять существующий код и предлагать рефакторинг. Агент работает синхронно в чате, но для длительных задач поддерживает фоновый режим с уведомлениями в VS Code. Все изменения применяются пользователем вручную — агент только предлагает код, патчи и скрипты.

Проблема:
При разработке сложных проектов (особенно с использованием LLM для генерации кода) разработчик и ассистент постоянно теряют контекст: забываются принятые архитектурные решения, структура проекта, зависимости между модулями, выбранные технологии. Это приводит к противоречивым рекомендациям, дублированию кода, ошибкам и потере времени.

Главная цель (SMART):
Создать агента, который к моменту завершения работы над проектом (через 2 месяца после старта разработки) не теряет связность контекста: все его рекомендации, генерации и объяснения остаются консистентными с историей проекта, даже если он был перезапущен или контекстное окно LLM было исчерпано. Агент должен уметь восстановить полную картину проекта из файлов Roadmap, чек-листа, лога изменений, Git-истории и графовой памяти.

Критерии успеха:

1. Агент всегда отвечает на вопросы о проекте, опираясь на актуальную структуру и историю, без противоречий с предыдущими решениями.
2. После перезапуска VS Code или смены модели агент может корректно продолжить работу над проектом с того же места.
3. При генерации кода для нового модуля агент использует уже существующие в проекте стили, библиотеки и архитектурные паттерны.
4. Разработчик (вы) отмечаете, что время на вспоминание контекста проекта сократилось как минимум на 70%.

2. Целевая аудитория и роли

Роль Описание Потребности
Разработчик-создатель (вы) Единственный пользователь агента. Создаёт различные проекты (мобильные приложения, веб-сервисы, SaaS). Обсуждение идей, генерация структуры и кода, объяснение кода, рефакторинг, написание тестов, анализ зависимостей, работа с Git-историей.
Администратор Та же роль — вы настраиваете модели, управляете памятью, редактируете профиль предпочтений. Гибкая настройка LLM, управление глобальным профилем, ручная корректировка памяти.

Ключевые пользовательские сценарии (User Stories):

1. Как разработчик, я хочу обсудить с агентом идею проекта в свободной форме, чтобы он помог формализовать требования и предложил структуру.
2. Как разработчик, я хочу получить от агента Roadmap и чек-лист проекта, чтобы иметь план и карту файлов.
3. Как разработчик, я хочу попросить агента сгенерировать код для модуля, чтобы он учитывал текущую структуру и технологии проекта.
4. Как разработчик, я хочу попросить агента объяснить непонятный фрагмент кода, чтобы быстро разобраться в чужом или старом коде.
5. Как разработчик, я хочу получить предложения по рефакторингу, чтобы улучшить качество кода без потери функциональности.
6. Как разработчик, я хочу, чтобы агент сгенерировал юнит-тесты для моего кода, чтобы ускорить тестирование.
7. Как разработчик, я хочу увидеть историю изменений файла через Git, чтобы понять эволюцию кода и причины решений.
8. Как разработчик, я хочу, чтобы агент находил все места в проекте, где используется определённая функция, чтобы безопасно проводить изменения.
9. Как администратор, я хочу подключить новую LLM через API-ключ и настроить системный промпт, чтобы использовать лучшую модель для конкретной задачи.
10. Как администратор, я хочу просмотреть и отредактировать графовую память проекта, чтобы исправить неверные связи.

# План реализации (поэтапно)

## Этап 1: Структурирование интервью (interview.json)

**Что делаем:**
1. Изменить `startInterview()` для создания `interview.json` вместо `interview.md`
2. Добавить интерфейс `InterviewData` с полями
3. Добавить валидацию обязательных полей
4. Обновить `handleRoadmap()` для чтения JSON

**Файлы:**
- `src/commands/CommandHandler.ts` — обновить `startInterview()`, `handleRoadmap()`
- `src/interfaces/IInterview.ts` — новый интерфейс

**Время:** ~1 час

---

## Этап 2: Стейт-машина для интервью

**Что делаем:**
1. Создать `.devil/.interview_status.json` при старте интервью
2. Обновлять статус при заполнении
3. Проверять статус перед генерацией roadmap
4. Добавить метод `getInterviewStatus()` в CommandHandler

**Файлы:**
- `src/commands/CommandHandler.ts` — добавить методы работы со статусом
- `src/interfaces/IInterviewStatus.ts` — новый интерфейс

**Время:** ~45 минут

---

## Этап 3: Улучшение `/checklist generate`

**Что делаем:**
1. Читать `roadmap.md` и извлекать этапы
2. Передавать этапы в промпт для LLM
3. Генерировать чек-лист с группировкой по этапам
4. Валидировать пути через `fileExists()`
5. Помечать существующие файлы как `[x]`

**Файлы:**
- `src/commands/CommandHandler.ts` — обновить `handleChecklist()`
- `src/utils/RoadmapParser.ts` — новый утилитарный класс для парсинга roadmap.md

**Время:** ~1.5 часа

---

## Этап 4: Команда `/checklist sync`

**Что делать:**
1. Добавить обработку команды `/checklist sync`
2. Сканировать реальную структуру проекта
3. Сравнивать с чек-листом
4. Обновлять статусы `[ ]` → `[x]`
5. Добавлять новые файлы
6. Генерировать отчёт

**Файлы:**
- `src/commands/CommandHandler.ts` — добавить `handleChecklistSync()`
- `src/utils/ChecklistSync.ts` — новый утилитарный класс

**Время:** ~1 час

---

## Этап 5: Команда `/roadmap update` (опционально)

**Что делаем:**

2. Перегенерировать roadmap.md с учётом новых данных
3. Сохранять предыдущую версию как `roadmap_v{N}.md`
4. Обновлять статус в `.interview_status.json`

**Файлы:**
- `src/commands/CommandHandler.ts` — добавить `handleRoadmapUpdate()`

**Время:** ~45 минут

1. Нефункциональные требования

Категория Требование
Производительность Максимальный размер проекта — до 1500 файлов. Для простых вопросов без сканирования — ответ мгновенный. Для сложных задач (сканирование, рефакторинг) — приоритет на качество, скорость вторична.
Синхронность/Асинхронность В основном синхронный режим (вы ждёте ответ в чате). Для длительных задач (более 1 минуты) возможен фоновый режим с уведомлением в VS Code.
Безопасность Все API-ключи хранятся локально, шифруются через защищённое хранилище ОС. Чувствительные данные (пароли, токены) в коде игнорируются агентом (не выводятся в ответах). Передача кода происходит только к выбранной LLM через прокси или напрямую.
Отказоустойчивость При ошибке соединения с LLM — автоматический повтор запроса с настраиваемым количеством попыток (по умолчанию 3). При превышении — понятная ошибка в чате.
Доступность Агент работает локально на вашем компьютере, не требует внешнего сервиса (кроме API LLM). Доступность = доступность вашей машины.
Платформы Только Windows 11, VS Code (последняя стабильная версия).
Accessibility Не требуется.

7. Данные и аналитика

Основные сущности

· Проект (путь, название, текущая модель, метаданные)
· Roadmap (Markdown-файл с планом)
· Чек-лист (Markdown-файл с путями и описаниями файлов)
· Лог изменений (журнал действий: что, когда, в каком файле)
· Граф памяти (узлы: файлы, классы, функции, технологии, решения; связи: вызывает, импортирует, использует, зависит)
· Диалог (история сообщений в чате по проекту)
· Глобальный профиль (стили кода, предпочтения, часто используемые библиотеки)

Хранение

· Глобальные настройки и профиль — %APPDATA%/Devil (Windows), шифрование API-ключей через защищённое хранилище ОС.
· Данные по проекту — в папке .devil/ в корне проекта (Roadmap, чек-лист, лог, графовая БД, история диалогов).
· Графовая память — SQLite (легковесная, встроенная БД) внутри .devil/.
· Кэш ответов LLM — опционально, SQLite для ускорения повторяющихся запросов.

Аналитика и отчётность

· Для MVP не требуется. В будущем — дашборд с метриками использования (количество запросов, сгенерированных строк кода, типов задач) — Won't have в первой версии.

Соответствие законодательству

· Поскольку данные хранятся локально и используются только вами, требования GDPR / 152-ФЗ не применяются.

9. План спринтов (с учётом использования AI-инструментов)

Общая длительность: 4 спринта по 2 недели (итого 8 недель), разбитых на 2 фазы.
AI-инструменты в разработке: GitHub Copilot (для автодополнения и генерации типовых конструкций), ChatGPT / Claude (для генерации сложных архитектурных решений, написания документации, рефакторинга кода расширения), автогенерация unit-тестов (Jest для TypeScript).
